**版本**: 1.0

---

## 🎯 核心原则

> **"标准化、可重复、主动汇报、持续改进"**

每次修复 BUG 都按这个流程来，保证质量一致、不遗漏步骤。

---

## 📋 完整流程（7 个阶段）

```
1. 问题接收 → 2. 分析定位 → 3. 修复实施 → 4. 测试验证
      ↓
7. 流程回顾 ← 6. 记录日志 ← 5. 提交代码
```

---

## 📝 阶段 1：问题接收

### 触发条件

小云描述 BUG 现象，例如：
- "归朴的 xxx 功能出问题了"
- "飞书通道报错 xxx"
- "模型调用失败"

### 小爪的行为

**1. 记录问题信息**
```
- 时间：2026-03-28 14:30
- 现象：xxx
- 影响范围：xxx
- 紧急程度：高/中/低
```

**2. 确认环境**
```
- 哪个通道：飞书 / Mattermost / WebUI
- 哪个模块：对话系统 / 工具调用 / 记忆系统
- 运行状态：服务是否正常运行
```

**3. 回复小云**
```
"收到，小云。我马上分析这个问题，请稍等。

问题记录：
- 现象：xxx
- 影响：xxx

开始定位根因... 🔍"
```

## 🔍 阶段 2：分析定位

### 小爪的行为

**1. SSH 到飞爪服务器**
```bash
ssh shangguanyun1@192.168.31.204
cd /归朴/return
```

**2. 查看日志**
```bash
# 查看最新日志
tail -f logs/xxx.log

# 搜索错误
grep -i "error" logs/*.log | tail -20

# 查看特定时间段日志
grep "2026-03-28 14:" logs/*.log
```

**3. 检查代码**
```bash
# 定位相关模块
find . -name "*.py" | xargs grep "xxx_function"

# 查看最近修改
git log --oneline -10
git show HEAD
```

**4. 复现问题（如需要）**
```
- 在飞书通道发送测试消息
- 触发问题功能
- 观察日志输出
```

**5. 找到根因**

分析维度：
- 代码逻辑错误？
- 配置问题？
- 依赖服务异常？
- 模型 API 问题？
- 网络/权限问题？

### 汇报模板

```
小云，问题定位到了：

📍 根因分析
- 问题位置：`src/xxx.py` 第 123 行
- 问题类型：代码逻辑错误 / 配置错误 / 依赖异常
- 具体原因：xxx

📊 影响范围
- 影响功能：xxx
- 影响用户：所有用户 / 部分用户
- 紧急程度：高/中/低

🔧 修复方案
- 方案 1：xxx（推荐，风险低）
- 方案 2：xxx（备选）
- 预计时间：10 分钟

开始修复吗？"
```

## 🛠️ 阶段 3：修复实施

### 小云确认后

```

```

### 小爪的行为

**1. 创建修复分支（可选，重大修复时）**

**2. 修改代码**
```bash

# 或使用 sed 快速修复
sed -i 's/old_code/new_code/g' src/xxx.py
```

**3. 本地语法检查**
```bash
python3 -m py_compile src/xxx.py
```

**4. 汇报**
```
✅ 修复完成

修改内容：
- 文件：`src/xxx.py`
- 改动：xxx → xxx
- 行数：+5 -3

开始测试验证... 🧪
```

```
- 测试异常情况

```

```
- ✅ 边界情况正常（如测试）
```

## 📦 阶段 5：提交代码

### 小云确认后

```
小云：提交
```

### 小爪的行为

**1. 检查变更**
```bash
cd /归朴/return
git status
git diff
```

**2. 生成 commit message**

格式：
```
<类型>: <简短描述>

- 问题描述
- 根因分析
- 解决方案
- 影响范围

小云的直觉/决策（如有）：
- 对问题的理解
- 对解决方案的判断
```

类型说明：
- `fix`: Bug 修复
- `feat`: 新功能
- `refactor`: 重构
- `docs`: 文档更新
- `chore`: 日常维护

**3. 执行提交**
```bash
git add .
git commit -m "fix: 修复 xxx 问题

- 问题：xxx
- 根因：xxx
- 解决：xxx
- 影响：xxx"
```

**4. 推送到裸仓库**
```bash
git push origin main
```

**5. 等待自动部署**
```
remote: 🦞 归朴自动部署中...
remote: ✅ 部署完成！
```

### 汇报模板

```
✅ 代码已提交

提交信息：
- commit: 77e46e6
- 信息：fix: 修复 xxx 问题
- 推送：已推送到裸仓库
- 部署：自动部署完成

需要更新修复日志吗？ 📝
```

## ⚠️ 注意事项

### 1. 紧急修复

**场景**：生产环境严重故障，需要快速修复

**调整**：
- 阶段 7（流程回顾）延后执行
- 阶段 6（记录日志）可以先简略记录，事后补充
- 优先保证修复和测试

**流程**：
```
问题接收 → 分析定位 → 修复实施 → 测试验证 → 提交代码
                                                        ↓
                    流程回顾 ← 补充日志 ← 紧急修复完成
```

### 2. 复杂 BUG

**场景**：根因复杂，需要多次尝试

**调整**：
- 阶段 2（分析定位）可能需要多轮
- 每轮分析后都要汇报进展
- 避免长时间沉默

**汇报节奏**：
```
"开始分析...（5 分钟后）"
"排除了 xxx 原因，继续...（10 分钟后）"
"找到疑似根因，验证中...（15 分钟后）"
"确认根因，开始修复"
```

### 3. 需要小云决策

**场景**：多个修复方案，需要小云选择

**处理**：
- 列出所有方案
- 说明每个方案的优缺点
- 给出推荐方案
- 等待小云确认后再执行
