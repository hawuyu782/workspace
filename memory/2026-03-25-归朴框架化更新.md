## 🎯 今日主要工作

### 1. 归朴框架化重构

**目标**：将归朴从旧版通道架构迁移到框架化架构

**核心改动**：
- 创建框架层 (`src/framework.py`) 统一管理状态
- 创建无状态核心 (`src/webchat_core_refactored.py`) 专注处理逻辑
- 重构飞书通道 (`start_feishu_framework_streaming.py`) 支持框架化调用

**架构变更**：
```
旧架构：
飞书通道 → WebChatCore (有状态) → 模型

新架构：
飞书通道 → 框架层 (状态管理) → 无状态核心 → 模型
```

**职责分离**：
| 层 | 职责 |
|------|------|
| **通道层** | 接收消息、发送回复 |
| **框架层** | 状态管理、唤醒流程、历史加载/保存、心跳、里程碑 |
| **核心层** | 消息处理、模型调用、工具执行 |

### 2. Bug 修复：MiniMax API 500 错误

**问题现象**：
- 飞书通道中时常出现 `MiniMax API 调用失败：500`
- 错误信息：`{"type":"server_error","message":"unknown error, 520 (1000)"}`

**问题根因**：
- 上下文太长（120 条历史对话）
- 超出了 MiniMax 服务器的处理能力
- 导致服务器返回 500 错误

**修复方案**：
```python
# 修改前
MAX_CONTEXT_TOKENS = 6000
MAX_HISTORY_MESSAGES = 60 条

# 修改后
MAX_CONTEXT_TOKENS = 5000
MAX_HISTORY_MESSAGES = 50 条  # 减少约 17%
```

**文件**：`src/framework.py`

### 3. 流式传输开发（暂停）

**尝试**：
- 集成旧版流式卡片模块 (`streaming_card.py`)
- 实现异步流式处理回调 (`process_message_callback_streaming`)

**发现问题**：
1. 流式函数定义重复（47 行和 285 行各有一个）
2. 异步事件循环冲突（`asyncio.get_event_loop()` vs `asyncio.new_event_loop()`）
3. 飞书官方 API 使用错误（应该用 Message API 而不是 CardKit API）
4. 更新频率超标（100 次/秒 vs 官方限制 10 次/秒）

**决定**：暂停流式传输开发，优先保证稳定性
