# Agent Teams 文档笔记

> 来源: https://code.claude.com/docs/zh-CN/agent-teams

## 概述

协调多个 Claude Code 实例作为一个团队一起工作，具有共享任务、代理间消息传递和集中管理。

> ⚠️ Agent teams 是实验性功能，默认禁用

## 核心概念

### 何时使用 Agent Teams

- **研究和审查**：多个队友同时调查问题不同方面
- **新模块或功能**：队友各自拥有独立部分
- **使用竞争假设进行调试**：并行测试不同理论
- **跨层协调**：前端、后端和测试各自负责

### 与 Subagents 比较

| | Subagents | Agent Teams |
| :-- | :-- | :-- |
| **Context** | 自己的 context window，结果返回给调用者 | 自己的 context window，完全独立 |
| **通信** | 仅向主代理报告结果 | 队友直接相互发送消息 |
| **协调** | 主代理管理所有工作 | 具自我协调的共享任务列表 |
| **最适合** | 结果重要的专注任务 | 需要讨论和协作的复杂工作 |
| **令牌成本** | 较低 | 较高 |

## 架构

| 组件 | 角色 |
| :-- | :-- |
| **Team Lead** | 创建团队、生成队友和协调工作的主会话 |
| **Teammates** | 各自处理分配任务的单独 Claude 实例 |
| **Task List** | 队友认领和完成的共享工作项列表 |
| **Mailbox** | 代理之间通信的消息系统 |

## 显示模式

- **In-process**：所有队友在主终端内运行，使用 Shift+Up/Down 选择队友
- **Split panes**：每个队友获得自己的窗格，需要 tmux 或 iTerm2

## 核心功能

### 消息传递
- **message**：向特定队友发送消息
- **broadcast**：同时发送给所有队友

### 任务管理
- 负责人创建任务，队友完成
- 三种状态：待处理、进行中、已完成
- 任务可以有依赖关系

### 委派模式
- 按 Shift-Tab 进入委派模式
- 负责人只能协调（生成、消息传递、关闭队友），不能直接实施

## 最佳实践

1. **给队友足够的 context** - 在生成提示中包含特定任务详情
2. **适当调整任务大小** - 自包含单位，产生清晰可交付成果
3. **等待队友完成** - 不要自己开始实施
4. **从研究和审查开始** - 不需要编写代码的任务
5. **避免文件冲突** - 每个队友拥有不同文件集
6. **监控和引导** - 定期检查进度

## 限制

- In-process 队友没有会话恢复
- 任务状态可能滞后
- 关闭可能很慢
- 每个会话一个团队
- 没有嵌套团队
- 负责人是固定的
